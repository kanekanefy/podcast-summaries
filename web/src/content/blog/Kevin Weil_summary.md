# Kevin Weil - 播客摘要

**嘉宾简介**: Kevin Weil - OpenAI首席产品官，前Instagram和Twitter产品负责人，Facebook Libra加密货币项目联合创始人

## 1. 核心议题

- **AI演进速度**: "你现在使用的AI模型是你余生中会用到的最差的AI模型" - 每2个月计算机就能做以前从未做到的事情
- **评估（Evals）作为核心技能**: 产品构建者需要掌握的全新手艺 - 为模型编写测试的能力
- **ChatGPT的消费者心智优势**: 如何在激烈的AI竞争中保持领先地位
- **迭代部署哲学**: 与社会共同进化的产品开发方式

## 2. 核心洞察

- **AI时代的技术基础根本不同**: "在任何其他地方工作过，你知道你在构建什么技术。在AI中完全不是这样...每两个月计算机就能做从未做过的事情，你需要完全不同地思考"

- **评估（Evals）定义**: "几乎就像模型的测验，测试模型对某组主题材料的掌握程度或对某组问题的回答能力"

- **模型准确性决定产品设计**: "如果模型60%正确，你会构建非常不同的产品 vs 95%正确 vs 99.5%正确"

- **智能的多维性**: "智能是非常多维的...一个模型可能擅长竞技编程，但不一定擅长前端编程或后端编程"

- **Chat接口的普遍性**: "聊天是一个惊人的接口，因为它是如此通用...它是我们交流的方式...你能够交流的事情多得多"

- **内部病毒式传播作为成功指标**: Instagram Stories和OpenAI ImageGen的共同点 - 内部使用爆发是产品成功的早期信号

## 3. 心法与原则

- **"AI是还没完成的事"**: 一旦能用了就被称为机器学习，一旦无处不在就只是算法 - "AI"总是指那些还不太工作的东西

- **计划无用，规划有用**: 艾森豪威尔名言 - 我们不认为3个月写下的东西就是我们真正要发布的，但规划时刻的价值在于停下来思考"我们做了什么？什么有效？什么没效？我们学到了什么？"

- **模型极简主义（Model Maximalism）**: "我们的通用心态是两个月后会有更好的模型，它会吹散当前的所有限制...我们不花太多时间为那些不匹配的部分搭建脚手架"

- **人类类比思维**: 设计AI体验时，思考"等效人类会怎么做" - 如果问我需要20秒思考的问题，我不会静音关闭然后20秒后回来

- **科技公司即模型集成**: "公司可以说是一个模型集成，所有这些模型都根据我们在大学学到的东西和职业生涯中学到的东西进行了微调"

## 4. 实战手册

**深度研究产品开发流程**:
1. 设计evals与产品并行开发
2. 创建英雄用例 - "这是一个你想问的问题。这是那个问题的惊人答案"
3. 将这些转换为evals
4. 在这些evals上爬坡
5. 通过微调让模型在这些用例上变得更好

**快速迭代文化**:
- 自下而上的团队授权
- 轻量级季度路线图 - "我们试图保持非常轻量级，因为它不会是对的"
- 允许犯错 - "Sam推动我们快速行动，但也理解快速行动意味着犯错"
- 糟糕的命名 - "o3 mini high" - 不是最重要的事情所以不花太多时间

**PM招聘标准**:
- **High agency（高自主性）**: "不会等着其他人允许他们做事情的人，他们会看到问题然后去做"
- **舒适地处理模糊性**: "对于更初级PM来说有时会困难...问题太不明朗了"
- **通过影响力领导**: "特别是研究功能更加自我导向"
- **果断性**: "PM需要能够在模糊存在且没有人做决定的地方确保做出决定"

**AI产品使用指南**:
- 使用ChatGPT总结文档、帮助写文档、GPTs写产品规范
- 使用模型帮助写evals - "它们相当擅长"
- Vibe coding（氛围编码）- 让模型主导编程，只需不断点击"是"
- 应该更多使用vibe coding制作原型而不是Figma设计

**模型集成策略**:
- 客户支持自动化 - 使用内部资源、知识库、指导原则教导模型
- 不同大小模型组合 - O系列用于推理，4o mini用于快速检查
- 细分问题 - "将问题分解为更具体的任务"
- 针对特定任务使用特定模型，包括微调

## 5. 反常识视角

- **Chat接口被低估**: 很多人认为"我们会找到比聊天更好的东西"，但"聊天是非常普遍的，因为它是我们交流的方式"

- **PM数量应该少**: "我个人认为组织应该PM相对较轻...太多PM会导致问题。他们会用decks和想法填满世界而不是执行"

- **招聘的9天沉默**：Kevin被OpenAI录取的经历 - 面试后9天没消息，因为内部太忙，他每天焦虑等待

- **Libra项目的失望**: "可能是我职业生涯最大的失望...如果我能推出那个产品，世界会变得更好。我能在WhatsApp免费给你发50美分。它应该存在"

- **写作能力的重要性**: (从Kevin Yien讨论中) 如果PM不能销售或支持自己的产品，我不信任他们构建产品
