# Teresa Torres 播客摘要

## 嘉宾简介

Teresa Torres 是产品发现领域的专家,她的书《Continuous Discovery Habits》在 Lenny 的 Slack 社区中始终是被推荐最多的第一本书。她可能是在全球范围内与产品经理合作、教学和影响人数最多的前五人之一。

她在产品管理社区中教授和推广了两个核心框架:
1. **Opportunity Solution Tree**(机会解决方案树)
2. **Continuous Discovery**(持续发现)

## 核心议题

- Opportunity Solution Tree 框架及其应用
- Continuous Discovery(持续发现)的理念和实践
- 如何自动化客户访谈流程
- 访谈技巧和常见错误
- 如何在组织中推动持续发现
- 假设测试与交付的关系

## 核心洞察

**Discovery vs. Delivery**:
- Discovery = 决定构建什么的工作
- Delivery = 实际构建的工作
- 传统的"先发现,后交付"思维是错误的
- 应该同时进行发现和交付

**Continuous Discovery(持续发现)**:
- 数字产品永远不会"完成"(done)
- 需要持续包含客户在决策过程中
- 建立持续反馈循环
- 可以通过每周一次访谈来实现

**Opportunity Solution Tree**:
- 从业务成果(Business Outcome)开始
- 分解为机会(Opportunities)
- 为每个机会探索解决方案(Solutions)
- 通过假设测试来评估解决方案
- **关键**:机会空间必须是需求/痛点,不是解决方案

**Assumption Testing ≠ Delivery**:
- 假设测试是交付的开始
- 两者是同一工作,没有明确的分界线
- 不应该把发现和交付看作分离的阶段

## 心法与原则

### 关于持续发现

**1. 数字产品没有"完成"的概念**

传统产品(实体商品):
- 设计 → 构建 → 制造 → 放在货架 → 完成 → 移到下一个

数字产品:
- Netflix 团队不会某天醒来说"嘿,我们的产品够好了"
- 总是在迭代,总是在改进
- 客户需求总是在演变
- 总有更多可以做的事情

**含义**:
- 从项目心态(project mindset)转向持续心态(continuous mindset)
- 持续做出关于构建什么的决策
- 因此需要持续包含客户在过程中

**2. "没有时间做发现"的真正含义**

当领导说"没有时间做发现"时,他们实际是说:
- "没有时间做基于项目的研究"(project-based research)

Teresa 的观点:
- 同意,我们确实没有时间做基于项目的研究
- 业务环境期望我们提供持续价值
- 需要匹配这个节奏(cadence)

**持续发现的优势**:
- 在访谈方面,每周一次访谈就可以实现
- 假设测试方面,假设测试和交付是同一工作
- 随着时间推移,你会做出更好的赌注(bets)

**3. 不要停止做赌注来开始发现**

常见的错误:
- "让我们停止做赌注,直到我们发现"
- 这是扼杀发现兴趣的最好方法

正确的方法:
- 继续做赌注
- 同时开始做一些发现
- 最终那些赌注会变得更好
- 发现和交付总是同时进行,不是先做这个再做那个

**4. 背景中的所有东西都是赌注**

- "无论我们是否做发现,一切都是赌注"
- 发现帮助我们做出更好的赌注
- 有时候需要做很多发现来做出尽可能好的赌注
- 有时候可以做有风险的赌注
- 如果你的所有赌注都是有风险的(因为零发现),最好的方法是在继续做赌注的同时并行开始做发现

### 关于机会解决方案树

**1. 核心结构**

```
Business Outcome(业务成果)
  ↓
Opportunities(机会 - 需求/痛点/愿望)
  ↓
Solutions(解决方案)
  ↓
Assumption Tests(假设测试)
```

**2. 为什么需要这个框架**

问题背景:
- 团队习惯于被告知"构建这些功能"
- 这是很结构化的"输出一些工作"问题
- 现在要求团队做根本不同的工作类型
- 团队需要一些脚手架(scaffolding)来做出转变

OST 的目的:
- 为这个广阔、混乱的问题增加一些结构
- 帮助团队可视化策略
- 帮助确定优先级
- 获得人们的认同
- 让每个人达成一致

**3. 机会空间的常见错误**

Teresa 看到的博客文章:
- "我会为这些糟糕的例子哭一点"
- 因为他们的机会空间全是解决方案
- 不想打击别人的博客文章
- 但也不想让这个坏例子在世界传播

**关键**:
- 机会必须是客户的需求、痛点、愿望
- 不是"我希望这更容易使用"(这是解决方案导向)
- 应该是"你在为谁解决什么?"
- 需要技能堆叠:好的访谈 + 能听到机会 + 能很好地框架它们 + 提取共同体验地图

**4. 成果导向思维**

传统公司:
- 产品团队被告知"构建这些东西"
- 不用担心我们为什么要做这些

OST 方法:
- 从业务成果开始
- 向上连接到业务成果
- 确保团队对成果负责
- 团队被赋予实现该成果的权力

### 关于访谈

**1. 访谈是被严重低估的技能**

"Grossly underestimated skill"

**2. 访谈的目标:收集故事**

大多数团队的错误:
- 问直接问题,脱离上下文
- "嘿 Lenny,你喜欢在 Netflix 上看什么?"
- "你如何决定看什么?"

正确的方法:
- 收集故事
- 理解客户行为的背景
- 理解他们实际做了什么

**3. 一句话就可以运行整个访谈**

Teresa 的示例:
- "Lenny,告诉我上一次你在流媒体娱乐服务上看东西的经历"
- 然后只需要:
  - "哦,你打开了 Netflix。接下来发生了什么?"
  - "哦,你没找到任何东西。怎么回事?"
- 只需要对体验好奇
- 问题只是帮助讲述时间线

**4. 访谈应该像和朋友喝啤酒**

- "如果你的访谈感觉像你在和 buddy 喝啤酒,那是好迹象"
- 应该那么随意、那么对话式
- 不会通过 pepper 你 50 个问题达到
- 通过收集故事、真正的好奇达到

**5. 我们变糟的原因**

- 太担心问下一个问题
- 停止倾听受访者
- 错过了被告知的一切
- 错过了"哦,有一些摩擦"的时刻
- 错过了"告诉我关于那个"
- 错过了"你在 Netflix 上考虑了什么?让我们深入挖掘"

**6. 客户喜欢好的访谈**

- "如果你在访谈中收集故事,客户会喜欢的"
- 好访谈的标志:客户会说"哇,我们什么时候可以再做一次?"

**7. 聚焦行为,不是想法/观点**

产品人的业务:
- 理解和改变行为

团队的错误:
- 在原型测试和访谈中
- 聚焦于人们会做什么
- 聚焦于人们想什么
- 聚焦于为什么他们认为他们做某事
- 都是真的很不可靠
- "垃圾进,垃圾出"的情况

正确的方法:
- 真正的衡量是告诉我你的行为
- 你实际上做了什么?
- 需要帮助人们做到这一点

## 实战手册

### 持续发现的自动化

**目标**:
让产品团队每周都有访谈,当他们周一早上醒来时,日历上已经有一个访谈,他们实际上什么都不用做。

**策略 1:产品内拦截(Product Interception)**

针对消费者和 B2B 最终用户:
- 几乎每个人都看过嵌入产品的 NPS 调查
- 相同的想法,但不是说"你会向朋友或同事推荐我们的产品或服务吗?"
- 而是说"你有 20 分钟和我们聊聊吗?"
- 如果他们说是,发送一些调度软件
- 他们选择你日历上的时间
- voila,你有一个预定的访谈

可以更高级:
- 在哪里展示?
- 展示给谁?
- 多少定制?
- 如何定位?
- 核心想法是让人们在他们已经有注意力的时候选择加入

**策略 2:使用内部团队**

针对买家和决策者:
- 相同的想法,但使用已经在和这些人通话的内部团队
- 销售人员、客户经理、支持人员
- 他们每天每个都在和这些人通电话
- 不使用产品来招聘
- 可以使用这些团队来招聘

实施方法:
- 团队每周定义一个触发器
- "嘿,这周我们想和经历这个需求或痛点的人交谈"
- 如果你碰巧和经历这个的人在通话
- 只需使用调度软件,把它放在我们的日历上
- 目标是产品团队根本不参与
- 他们只需要出现并连接访谈

**工具推荐**:
- 调度:Calendly、Outlook、Google 工具、Salesforce 工具
- 拦截:Qualaroo、Ethnio、Intercom、Usabilla、Chameleon、Hotjar

### Opportunity Solution Tree 构建步骤

**步骤 1:从业务成果开始**

- 明确你要驱动的业务成果
- 这是树的根

**步骤 2:识别机会**

机会从哪里来?
- 从客户的故事中涌现
- 需要好的访谈技能来收集丰富的故事
- 需要能够在访谈中听到机会
- 需要能够很好地框架它们
- 需要能够从看似独特的故事中提取共同的体验地图

**步骤 3:为每个机会探索解决方案**

- 不要只为一个机会探索一个解决方案
- 为同一个机会处理多个解决方案
- 这是防止你认为"明显解决方案"的最好方法

**步骤 4:假设测试**

- 评估解决方案
- 从非常具体的假设开始
- 运行多个假设测试
- 每周可以运行半打到一打假设测试
- 这些假设跨越三个想法
- 在周末,可以开始比较和对比这些解决方案

### 可持续的持续发现

**可持续性的关键**:

1. **改变方法,不是增加负担**
   - 不是在交付之外增加发现
   - 假设测试是交付的开始
   - 两者是同一工作

2. **小规模开始**
   - 每周一次访谈
   - 不要试图做太多
   - 建立习惯

3. **快速假设测试**
   - 那些挣扎着测试单个想法的团队仍然卡在基于项目的研究世界
   - 他们运行的实验需要数周才能得到结果
   - Teresa 谈论的假设测试:团队在一周内运行半打到一打假设测试
   - 这些假设跨越三个想法

4. **同时处理三个想法**
   - Teresa 告诉人们同时处理三个想法
   - 团队挣扎着测试甚至单个想法
   - 可持续性在于改变行为、改变习惯

### 当你已经知道解决方案时

**问题**:当你知道解决方案应该是什么时,如何保持纪律并保持开放思维?

**Teresa 的回答**:

1. **不是所有解决方案都需要大量发现**
   - 常见的误解
   - 需要对核心产品体验或将成为差异化的东西做真正强大的发现
   - 不需要对忘记密码流程做真正好的核心发现(如果它工作正常且你没听到它是痛点)

2. **评估赌注的风险**
   - 我们在做赌注
   - 这个赌注涉及多少风险?
   - 我们需要减轻多少风险?

3. **大多数公司认为所有赌注都没有风险**
   - 做零发现
   - 如果不工具化产品并实际测量解决方案的影响
   - 可能没有捕捉到实际上有很多风险

4. **需要工具化产品**
   - 需要测量发布的所有东西的影响
   - 可以开始磨练你对想法中风险的判断

5. **发现新手**:过度指数做一点太多的发现
   - 开始磨练你对风险的判断

6. **核心机会**:使用多个解决方案
   - 如果你在处理对你的产品功能真正核心的机会
   - 它是一个差异化因素
   - 你想确保你有一个真正强大的好的解决方案
   - 防止你认为明显解决方案的最好方法是为同一个机会处理多个解决方案

### 团队规模无关

**理想世界**:
- 如果我有一个 trio(三人组)
- 他们有成果
- 他们被赋予实现该成果的权力
- 他们每周访谈
- 他们的假设测试来评估解决方案
- 他们发现要构建的东西
- 他们驱动他们的成果

**这是一个非常成功的团队**:
- 可以在三人公司做到
- 可以在十万人的公司做到

**关于样本量的常见反驳**:

"基于一次访谈做决定可靠吗?"

Teresa 的反驳:
- "告诉我你上周做的决定"
- "你和多少客户谈过?"
- "你使用了什么数据?"
- 为什么产品团队突然被其他人都没有遵守的标准所束缚?

### 两个核心活动

**从发现的角度**:

1. **定性访谈**
   - 目标:了解机会空间
   - 我们在哪里看到未满足的需求、痛点和愿望?
   - 访谈不是识别机会的唯一方法
   - 观察(observation)实际上是更好的方法
   - Teresa 专注于访谈是因为我们可以每周可持续地做
   - 大多数团队没有能力每周观察他们的客户

2. **假设测试**
   - 任何帮助我们评估解决方案的东西
   - 从非常具体的假设开始
   - 不应该运行的一些实验:
     * 在有任何想法是否有强大基础的想法之前测试整个想法
     * 太长
     * 成本太高
     * 太多时间
   - 需要分解

## 反常识视角

### 1. "先发现,后交付"是错误的

大多数人认为:
- 先做发现
- 然后做交付

Teresa 的观点:
- 你总是在交付
- 你总是在发现
- 假设测试是交付的开始
- 两者之间没有明确的界限
- 不要把它看作阶段

### 2. 访谈应该是随意的,不是正式的

常见的错误:
- 50 个问题的访谈协议
- 问 who, why, how 的问题
- 导致不是自然对话的节奏

Teresa 的方法:
- 如果你的访谈感觉像你在和 buddy 喝啤酒,那是好迹象
- 一句话就可以运行整个访谈
- 只需要对体验好奇

### 3. 客户喜欢访谈(如果你做得对)

常见的担忧:
- "我不想向他们请求恩惠"
- 销售代表说"我不想问客户"

现实:
- 如果你收集故事,客户喜欢它
- 好访谈的标志:客户会说"哇,我们什么时候可以再做一次?"

### 4. 一次访谈足以

人们质疑:
- "基于一次访谈做决定可靠吗?"

Teresa 的反驳:
- 告诉我你上周做的决定
- 你和多少客户谈过?
- 你使用了什么数据?
- 为什么产品团队被其他人都没有遵守的标准所束缚?

### 5. 不要停止做赌注来开始发现

常见的错误:
- "让我们停止做赌注,直到我们发现"

正确的方法:
- 继续做赌注
- 同时开始做发现
- 最终赌注会变得更好
- 这是扼杀发现兴趣的最好方法

### 6. 机会空间不是解决方案空间

常见的错误:
- 把"我希望这更容易使用"当作机会
- 这是解决方案导向

正确的方法:
- "你在为谁解决什么?"
- 机会必须是需求、痛点、愿望
- 需要技能堆叠:好的访谈 + 能听到机会 + 能框架它们 + 提取共同体验
