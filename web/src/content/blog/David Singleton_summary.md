**嘉宾简介**: David Singleton - Stripe CTO，前 Google VP 工程，长期负责大规模产品与工程文化建设。

1. **核心议题 (The Questions)**
   - **Stripe 为什么能长期吸引并留住顶级人才**？
   - **“在工艺上极度认真”如何被制度化**而不是口号？
   - **如何在高可靠性与高速迭代之间同时做到**？
   - **AI 对 Stripe 产品与研发工作方式的影响**在哪里？

2. **核心洞察 (Core Conclusions)**
   - **用户共创是 Stripe 的核心研发模型**：Billing 的早期功能由 Figma、Slack 等真实用户共同塑形。
   - **“认真打磨”的价值来自可量化收益**：支付结账优化带来 **+10.5% 收入提升**，细节能产生复利。
   - **高可靠≠低变更频率**：Stripe 通过自动化测试与灰度发布实现 **45 分钟进生产**的高频发布。
   - **AI 价值首先体现在“降低理解与表达门槛”**：文档问答、自然语言写 SQL、内部提示词共享让非工程角色也能高效使用。

3. **心法与原则 (Repeatedly Emphasized Points)**
   - **用户第一[users first]**是所有流程的起点，不是价值观口号。
   - **高标准来自可复用实践**：摩擦记录[friction logging]、UX 评审、Walk the Store 等机制固定化。
   - **信任先行再验证**：招聘时更重视引用背书，假设新人值得信任，然后用结果去校准。
   - **时间与精力管理是高管刚需**：每周设定“成功清单”，否则时间被日程随机吞噬。

4. **实战手册 (Specific Cases & Methodologies)**
   - **摩擦记录[friction logging]**：选择具体用户画像，完整体验流程，记录痛点与“亮点”，并定期复盘。
   - **工程假期[engineerication]**：CTO 亲自加入团队 3–4 天游泳式交付，输出摩擦报告。
   - **协作式 UX 评审**：跨职能集体走流程、共享问题清单、会后集中定优先级。
   - **持续交付链路**：自动化测试 → 多环境验证 → 小流量灰度 → 全量发布。
   - **事故处置体系**：即时响应 + 深度复盘 + 类别级预防，优先级高于新功能。
   - **规划流程“倒 W”**：团队自下而上提案 → 领导层综合 → 下放反馈 → 再综合。

5. **反常识视角 (Counter-intuitive Points)**
   - **金融基础设施也可以“分钟级发布”**，前提是系统化测试与分层灰度。
   - **CTO 需要“亲自写代码”**不是为了输出，而是为了真实理解工具链与摩擦点。
   - **AI 推进不靠政策宣导**，而是通过“内部可用工具 + 共享提示词模板”降低门槛。

6. **花絮与细节 (Interesting Details)**
   - **术语**：Stripe 员工自称 **“Stripes”**。
   - **经典细节**：API 报错直接链接对应文档，**错误处理代码比主流程还多**。
   - **推荐书**：《High Output Management》《Build》《Scaling People》。
   - **AI 应用**：GPT-4 文档问答、Sigma 自然语言写 SQL、内部安全版聊天工具与提示词库。
