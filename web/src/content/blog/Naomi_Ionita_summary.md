# Naomi Ionita - Lenny's Podcast 摘要

## 嘉宾简介

Naomi Ionita是Menlo Ventures的合伙人，专注于早期SaaS投资（从种子轮到B轮）。她是产品生命周期增长和货币化的早期领导者之一，十多年前就在Evernote建立了早期团队和基础设施。她也是Reforge的早期贡献者，帮助创建了早期项目。她曾是Invoice2go的增长副总裁，目前是全职VC。

## 核心议题

1. **货币化的三大错误** - 等太久、定价过低、设置就忘
2. **定价与价值匹配** - 寻找正确的价值指标
3. **Day 1 vs Day 100功能** - 简化定价包装的思维框架
4. **现代增长栈** - 数据驱动增长的新工具生态

## 核心洞察

### 1. 货币化的三大常见错误

**错误一：等待太久才开始收费**
```
创业初期的重点:
→ 创造有价值的东西
→ 创始人有独特的市场洞察
→ 对痛点的真实性
→ 新颖的解决方案
→ 改变世界

硬币的另一面:
→ 作为企业获得适当的补偿
→ 真正的产品市场契合信号
→ 是人们愿意为你打开钱包

不要等太久:
→ 从构建产品转向构建业务
→ 不要无限期地延迟货币化
→ 免费beta用户反馈循环是R&D成本
→ 确保你在构建最好的产品
```

**不收费的代价**:
```
你在伤害自己:
1. 无意中贬低了你的产品
   → 人们给你的构建物赋予较低的美元价值或0美元价值

2. 错过关键反馈循环
   → 不了解人们愿意付什么钱

3. 为未来的自己埋下隐患
   → 某个时候你会开始收费
   → 会经历一些抵制
   → 最好提前做好准备
```

**错误二：定价过低**
```
表现:
→ 不仅基础价格太低
→ 通过不提供不同的计划来迎合不同的细分市场
→ 把钱留在桌子上

Evernote的例子:
→ 从一开始每年$45
→ 为公司奠定了基础
→ 产生了数千万美元的早期收入增长
→ 但这种方法是次优的

调查发现:
→ 人们从免费版转换为高级订阅的最受欢迎答案之一是：
   "我只是感到内疚。我使用它太多了。
    我从中获得如此多的价值，
    以至于我只是觉得有义务付费。"

如果内疚是人们付费的主要原因之一:
→ 那么你的免费版太好了
→ 你把钱留在桌子上
```

**单一高级层级的错误**:
```
Evernote的两种用户类型:

1. 全新用户（低感知价值）:
→ 看待Evernote就像设备上预装的Apple Notepad应用
→ 无法理解为Evernote支付$45

2. 狂热用户（高感知价值）:
→ 跨客户端使用（桌面和移动）
→ 用于工作和个人
→ 利用OCR功能和网页剪辑器
→ 真正的"第二大脑"
→ 无法想象没有它的生活
→ 对他们只付$45感到震惊
→ 说他们从Evernote获得数百美元的价值

解决方案:
→ 针对不同角色制定不同计划
→ 基于他们从产品中获得的价值
→ 以及他们的支付意愿
```

**错误三：设置就忘**
```
不要设置就忘记:
→ 我看到公司这样做
→ 他们精心设计和开发功能
→ 构建这个完美的产品，使用起来很愉快
→ 然后定价是从空中摘出来的
→ 然后他们不再重新审视

Evernote的情况:
→ 过了很多很多年
→ 我们才回去改革定价

建议:
→ 像对待路线图一样对待你的定价
→ 每6到12个月
→ 你可能正在为用户推出有意义的东西
→ 把这视为重新审视你的货币化策略的机会
→ 确保你得到适当的补偿
```

### 2. 定价与价值匹配

**核心概念**:
```
匹配价格与价值:
→ 当你这样做时，你与用户创建了一致性
→ 这涉及选择正确的价值指标
→ 这是从使用你的产品中得出的价值单位
→ 它创造了一个自然的升级器
→ 随着人们更多地使用它，你会随着时间的推移得到更多报酬
```

**SaaS定价的演变**:
```
历史上的SaaS:
→ 基于座位的模式
→ 这是历史SaaS定价

PLG的兴起:
→ 看到更多这些基于使用的方法获得速度
→ 无论是API调用数量
→ 发送的消息数量
→ 使用的存储TB数量
→ 编写的字数

基于使用的方法:
→ 在客户生命周期内匹配价格与价值
→ 帮助你了解你在为谁构建
→ 让你能够针对不同的客户细分
→ 你能够更好地服务每个细分市场
→ 你也能够为企业最大化收入
```

**Envoy的10倍定价故事**:
```
场景:
→ Larry（Envoy创始人）
→ 与一家大型酒店公司会面
→ 对话进行得非常顺利
→ 潜在客户真的很兴奋

转变到定价时:
→ Larry因为感觉良好的氛围
→ 决定将价格提高10倍
→ 他通常向人们收取的价格

结果:
→ 高管说："好的，当然。听起来不错。"
→ 没有犹豫一分钟
→ 没有犹豫一秒钟

学到的教训:
1. 他明显定价过低
   → 他甚至没有考虑上限是什么

2. 他实际上可能可以推得更高
   → 考虑到没有犹豫

建议:
→ 尤其是在这些企业对话中
→ 继续要求更多
→ 了解上限可能在哪里
→ 理解由于价格失去一些交易是可以的
→ 20-30%左右是合理的
→ 这样你就可以了解限制可能在哪里
→ 绝大多数公司绝对收费过低
→ 像Envoy的Larry一样出去冒险
→ 有时你可以2X、4X甚至10X你的价格
```

### 3. Day 1 vs Day 100功能框架

**Invoice2go的实验**:
```
需求曲线的典型情况:
→ 价格越高
→ 每用户平均收入(ARPU)越高
→ 转化率越低
→ 这些东西成反相关

他们做的:
→ 重新平衡定价和包装
→ 实际上将从入门计划升级到专业计划的升级率翻了一番
→ 同时将专业计划的价格提高了约30%

结果:
→ 让更多人升级而支付更多
→ 这种复合好处相当罕见
```

**如何做到的**:
```
Day 1高级功能:
→ 可以从第一次使用产品时获得价值的功能
→ 这是高级功能
→ 与你的Day 100功能不同

Day 100功能:
→ 代表更高级的功能
→ 可能是在平台中拥有一定规模数据后才获得价值的功能
→ 你不应该浪费用户的认知负荷
→ 让他们在刚开始时甚至要理解或尝试欣赏
→ 把这些推到产品更高级的专业版本
→ 随着时间的推移通过追加销售来货币化
```

**简化定价**:
```
我们都见过那些SaaS定价页面:
→ 功能和功能清单
→ 或仅仅是复杂的矩阵

建议:
→ 尽你所能思考用户的旅程
→ 他们将如何随着时间的推移继续增加产品价值
→ 你如何根据那个旅程映射你的定价和包装
→ 单人模式 → 多人模式 → 团队 → 企业
→ 思考整个用户生命周期的纵向视图
→ 整体地思考你的商业模式
```

### 4. Freemium策略

**Naomi的立场**:
```
这不意味着我不喜欢Freemium模式:
→ Evernote十年前是Freemium的宠儿
→ 我仍然是它的忠实信徒
→ 更多是你在哪里放置付费墙的问题
→ 你免费提供多少？
→ 然后你如何定价和包装产品的付费版本？
```

**Freemium的目标**:
```
Freemium是为了:
→ 获得漏斗顶端的兴奋
→ 让人们养成习惯
→ 缩短到价值的时间
→ 建立习惯形成
→ 建立所有这些冠军来使用你的产品

但想法是:
→ 将他们引导到产品的付费版本
→ 再次不延迟"我们的高级功能应该是什么？
   付费计划应该是什么样？"的想法
```

**什么应该是免费的？**:
```
如果它能让你达到"啊哈"时刻:
→ 那条通往习惯形成的道路必须是免费的

核心效用:
→ 那是你的产品的核心效用

目标:
→ 在第一次会话或第一天
→ 某人看到快乐并说
→ "天哪，我再也不会回到旧的方式了。
   这就是X完成的方式。"

网络效应:
→ 如果你在寻找病毒式传播或网络效应
→ 这是另一件事
→ 你的免费用户可能不会从你那里获得收入
→ 但想法是他们帮助你管理CAC
→ 这些人为你推动有机增长
→ 帮助降低下一组用户的增量成本
→ 这是在放弃收入的数学方程中要考虑的另一部分
```

### 5. 定价变化的影响

**惊人的数据**:
```
OpenView的研究:
→ 实施定价变化的公司中
→ 大约一半看到至少25%的ARR增长
→ 这是一个相当巨大的阶跃式改进
→ 来自不需要大规模技术革新的东西

ProfitWell的调查:
→ 500多家SaaS公司
→ 查看1%的获取、留存和货币化改进
→ 如何影响公司的底线？
→ 发现货币化改进的影响
→ 是获取的4倍

结论:
→ 如何有效地改善你的业务货币化
→ 真的被低估为一个增长杠杆
→ 绝对是人们应该考虑的事情
→ 我做这个播客的部分目标
→ 是确保创始人得到他们应得的补偿
```

**Naomi见过的最大影响**:
```
我看到收入向上提升10倍:
→ 但很难仅仅分离定价变化
→ 因为它通常与重大的产品变化结合
→ 重新品牌
→ 大量PR
→ 新计划的发布
→ 团队或企业计划

Envoy的影响:
→ 在那一刻学到了
→ 1. 他明显定价过低
   → 他甚至没有考虑上限是什么
→ 2. 他实际上可能可以推得更高
   → 考虑到没有犹豫
→ 这是一个具体的例子
→ 显示定价变化可以立即产生10倍影响
```

**定价变化的成功要素**:
```
它来自整体思考:
→ 我认为如果你只是选择一个新价格
→ 或者只是推出一个新计划
→ 很少有那么有影响力

重新平衡定价和包装:
→ 理解人们真正想要什么
→ 他们的支付意愿是什么
→ 将其映射到用户旅程
→ 就像我们谈论的从单人模式到多人模式
→ 你连接的第一个人，你有工作流程
→ 将它传播给你的整个团队
→ 最终在整个组织中传播

纵向视角:
→ 用户生命周期的纵向视角
→ 整体地思考你的商业模式
```

### 6. 如何确定初始价格

**完整的定价流程**:
```
第1步：组成定价委员会:
→ 这不是一个单线程的练习
→ 不存在于一个部门或另一个部门
→ 这是一个跨职能的练习

PLG公司:
→ 如果你是像我工作的公司那样的PLG公司
→ 产品增长组织
→ PM和数据科学家的组合
→ 像这样的人来迭代定价

企业SaaS业务:
→ 销售、财务和RevOps发挥作用
→ 思考那个委员会应该在你公司是谁
→ 致力于成为真正拥有并随时间迭代定价的跨职能团队
```

```
第2步：与客户交谈:
→ 这是到目前为止最基本的事情
→ 你可以做的只是增加那些反馈循环
→ 了解你可以在定价上推动多少

方法:
→ 调查
→ 访谈

问题示例:
→ 了解功能的相对优先级
→ 回到定价页面上的功能清单和矩阵
→ 人们很少在所有这些功能上平等转换
→ 通常有一两个是转换的主要点

100分问题:
→ 给用户100分
→ "把这些分数分配给不同的功能"
→ 某个功能给的分数越多
→ 你赋予它的价值就越多

问题分类:
→ 必须有
→ 最好有
→ 不需要

这有助于你:
→ 理解相对优先级
→ 如何调和一些定价和包装
```

```
第3步：Van Westendorp方法:
→ 如果你获取用户指定为"最好有"和"必须有"的软件包
→ 然后问他们：

1. "什么样的便宜价格开始让你质疑产品的质量？"
2. "什么样的好交易或听起来像是这个软件包的正确价格？"
3. "什么是昂贵的，但他们仍然会付钱？"
   → 你开始达到那种不适的程度

4. "什么是禁止性的昂贵？
   人们只会说，'好吧，就是这样。'
   你已经跨越了我愿意在这里支付的界限。"

通过绘制这四条曲线:
→ 你开始了解如何为你的定价提供信息
→ 这是一个很好的方法
→ 将关于需求的问题与关于支付意愿的问题结合起来
```

### 7. 测试定价变化

**为什么很难测试定价**:
```
有很多不同的变量需要隔离:
→ 你必须为测试带来一致的体验
→ 产品内体验
→ 定价页面
→ 移动应用商店
→ 你发送的生命周期电子邮件

技巧:
→ 按地理位置细分这些测试
→ 我们会在加拿大或澳大利亚进行一些测试
→ 然后在美国推出
→ 这是在实验周围施加一些约束的好方法
```

**定价实验的长期性质**:
```
了解你是否成功和失败:
→ 通常需要了解对流失的影响

示例:
→ 你的测试的一部分是第一年折扣
→ 你需要了解用户在第二年的表现
→ 有一种关于用户增长、留存、ARPU之间权衡的感觉
→ 这些都是你想随时间优化的不同杠杆
```

**Invoice2go的投资**:
```
他们在内部计量和人工管理:
→ 我们在内部计量和人工管理以及实验系统投入了大量资金
→ 这是一个巨大的增长工程事业

有了那个:
→ 我们测试了不同的价值指标
→ 测试了不同的配额限制
→ 价格点
→ 促销
→ 你能说出名字的东西

我们跟踪了按需付费模式的消耗:
→ 并将其循环回产品中
→ 这样我们就可以推动用户沿着生命周期
→ 让他们在达到配额限制时转换或升级或续订

那项投资非常值得:
→ 通过能够以更精简的方式迭代
→ 我们获得了巨大的收入收益
```

### 8. 现代增长栈 (Modern Growth Stack)

**什么是现代数据栈**:
```
云原生工具集合:
→ 更容易地移动和管理数据

组成部分:
1. 完全托管的ELT、数据管道
2. 数据目的地（基于云的数据仓库，如Snowflake或Redshift）
3. 数据转换工具（如DBT）
4. 可视化平台（人们可以访问数据）
```

**现代增长栈**:
```
现代数据栈的戏仿:
→ 非常有意的
→ 我的核心论文领域现在在Menlo
→ 是你用数据做的事情的演变
→ 这些是数据驱动业务向前的工作流
→ 针对像我过去运行的产品增长和收入团队

现代替代品:
→ 替代像我这样的团队构建或购买的基础设施
→ 当你负责推动激活或货币化或留存等事情时
→ 有很多这些内部工具
→ 因为你真的为跨职能团队提供动力
→ 它不容易映射到传统的部门
```

**三大核心主题**:

**1. 数据**:
```
智能集成和自动化:
→ 现代增长栈公司真的由智能集成和自动化驱动

反向ETL公司（如Hightouch或Census）:
→ 打破这些孤岛
→ 帮助公司或员工跨公司访问数据
→ 并提高生产力
```

**2. 工作流**:
```
人员和流程的赋能:
→ 员工不再坐在他们的部门孤岛中
→ 现代增长栈公司在他们之间建立桥梁

通过解锁数据访问:
→ 业务方面通常可以自助服务
→ 更加自给自足
→ 不依赖工程师或数据科学家来运行查询
→ 或为他们拼接数据集

跨职能工作:
→ 很多增长工作本质上是跨职能的
→ 驱动增长的努力需要新工具和协作工作流

没有专用软件:
→ 很多像我这样的团队觉得别无选择
→ 只能在内部构建
→ 我们花了几个小时来构建和维护工具
→ 用于实验、个性化、计费、货币化
→ 这些资源本来可以重新分配
→ 为企业构建专有功能
→ 如果我们能够买到专用的东西
```

**3. 影响**:
```
推动硬ROI:
1. 成本降低形式:
   → 自动化意味着节省时间
   → 通常可以直接映射到公司的成本降低

2. 收入影响形式:
   → 帮助产品和增长以及进入市场团队
   → 更好地参与和货币化客户

双重ROI故事:
→ 我对能够在这两个方面都推动硬ROI的公司感到非常信服
→ 在宏观经济较软的气候中
→ 这个ROI故事甚至更令人信服
→ 你必须能够继续保留销售和定价权
→ 我认为这来自于像我描述的那样强大的ROI故事
```

**现代增长栈的层次**:

**第1层：产品导向销售 (Product-Led Sales)**:
```
概念:
→ 为PLG业务服务的公司
→ 利用所有产品使用数据的力量
→ 告诉面向客户的团队
→ 哪些账户最可升级

这是免费的金钱:
→ 当你在一个没有人关注的账户上发出光时
→ 一些内部销售团队可以推动大型账户扩展

公司例子:
→ Endgame（有Figma、Loom、Calendly等客户）
→ Pocus（在构建内容和社区方面做得非常出色）
```

**第2层：实验**:
```
为什么这是关键基础设施:
→ 对于这些跨职能产品数据增长团队
→ A/B测试假设并了解其对业务的影响
```

### Van Westendorp定价方法的四个问题
```
问题1 - 太便宜:
"什么样的便宜价格开始让你质疑产品的质量？"

问题2 - 好交易:
"什么样的好交易或听起来像是这个软件包的正确价格？"

问题3 - 昂贵但可接受:
"什么是昂贵的，但他们仍然会付钱？"
→ 你开始达到那种不适的程度

问题4 - 太贵:
"什么是禁止性的昂贵？"
→ "你已经跨越了我愿意在这里支付的界限。"

使用方法:
→ 获取用户指定为"最好有"和"必须有"的软件包
→ 然后问这四个问题
→ 绘制四条曲线
→ 开始了解如何为你的定价提供信息
```

## 反常识视角

### 1. 内疚作为付费驱动因素
**Evernote的调查发现**:
- 人们从免费版转换为高级订阅的主要原因之一是："我只是感到内疚。我使用它太多了。"
- 如果内疚是主要驱动因素，说明免费版太好了
- 这意味着定价错误

### 2. 单一高级层级的错误
**很多公司**:
- 只有一个付费层级
- 为不同细分市场留钱在桌子上
- 需要深入挖掘并了解这些是谁
- 根据他们获得的价值和支付意愿制定不同的计划

### 3. 定价变化的巨大影响
**数据显示**:
- 大约一半实施定价变化的公司看到至少25%的ARR增长
- 货币化1%改进对底线的影响是获取1%改进影响的4倍
- 定价变化不需要大规模技术 overhaul
- 但可以产生阶跃式改进

### 4. 混合定价模型的优越性
**纯使用量模型**:
- 只有不到10%的公司使用
- 大多数使用混合方法（订阅+使用配额）

**为什么混合更好**:
- CFO想要可预测性
- 他们希望能够为你的工具做预算
- 但你仍然需要为他们交付的价值获得适当的补偿
- 平衡固定和可变组件

### 5. PLG公司的收入优化时机
**Figma的例子**:
- 花了一段时间才货币化
- 个人级别免费使用
- 目标是推动社区和对产品的爱
- 知道设计本质上是协作的
- 一旦嵌入企业，触发支付的开关

**教训**:
- 如果你知道你有向上市场的桥梁
- 放弃长尾个人用户可能是值得的
- 但不要让免费版太好了

### 6. 现代工具的ROI故事
**双重ROI**:
```
成本节约:
→ 自动化意味着节省时间
→ 通常可以直接映射到公司的成本降低

收入影响:
→ 帮助产品和增长以及进入市场团队
→ 更好地参与和货币化客户

在宏观经济较软的气候中:
→ 你必须能够继续保留销售和定价权
→ 这来自于强大的双重ROI故事
```

**核心启示**: Naomi Ionita的核心洞见在于：(1) 货币化应该像产品路线图一样持续演进，而不是"设置就忘记"；(2) 定价必须与价值匹配，通过Day 1 vs Day 100框架简化包装决策；(3) 现代增长栈通过数据、工作流和影响力三大主题，为增长团队提供了强大的工具集，将过去需要内部构建的复杂功能转化为现成可用的服务；(4) 混合定价模型（订阅+使用配额）通常比纯基于座位或纯使用量的模型更优；(5) PLG公司应该平衡增长与收入，利用产品使用数据驱动销售和扩张。最重要的是，货币化改进的影响通常是 Acquisition改进影响的4倍，是一个被严重低估的增长杠杆。
