# Kevin Yien - 播客摘要

**嘉宾简介**: Kevin Yien - Stripe商户体验产品负责人，前Square餐厅业务和生态系统团队负责人，前Mutiny产品和设计负责人，前竞争进食者，自制冰淇淋爱好者

## 1. 核心议题

- **PM角色定义**: 将团队的潜在能量转化为尽可能多的实现价值
- **为什么不要直接进入PM**: 从工程师、设计师或销售开始的职业路径
- **写作能力作为PM核心技能**: 清晰度规模化 = 写作能力
- **决策日志（Decision Log）**: 提升"产品感觉"的练习方法
- **边界绘制（Perimeter Drawing）**: PM为团队创造约束的艺术

## 2. 核心洞察

- **PM不应该直接做PM**: "如果你是客户，为什么需要别人帮你做决定...但如果你不是客户，如果你在特别复杂的领域工作，如果你觉得作为能够构建产品的人缺乏某些东西，那就是你可以委托责任的时候"

- **写作是清晰的规模化**: "PM工作的一个关键组成部分是创造清晰度，既对内也对外...如果他们不能销售或支持自己的产品，我不信任他们构建产品"

- **产品感觉就是数据不足时的决策能力**: "我们都谈论产品感觉。对我来说，这只是一种花哨的说法，表示你能在数据不足的情况下做出好的决定"

- **PM画边界，团队填满空间**: "PM应该尽一切努力画出问题空间的边界...在那个边界内，工程、设计、其他你与之合作的人可以尽可能疯狂，推到边界，最大化填满盒子"

- **最佳PM:工程师:设计师比例**: "我宁愿雇佣增量设计师而不是PM，几乎一周中的任何一天...3个工程师、3个设计师、1个PM，这是最好的比例"

- **调音叉（Tuning Fork）反馈法**: "获得反馈的最佳方式不是问他们怎么想，而是把具体的东西放在他们面前，然后让他们对它做出反应"

## 3. 心法与原则

- **潜在能量vs动能能量**: 高中科学的潜能和动能概念 - PM的作用是将团队的潜能转化为实现价值

- **不要妥协最终交付**: "你需要痴迷于最终交付以及价值是否真正到达客户...即使工程师会比你构建更好的产品，设计师会设计比你更好的产品，你需要带着强烈的意见来"

- **移动慢以移动快**: 通过实时会议审阅文档来减少反馈延迟 - "当你几乎同时评论和回应时，这确实能非常快地推动那个周期速度"

- **每日日志实践**: 受"Big Ass Text File"启发 - 一天中发生的一切都在一个大胆列表中，这样你可以command F搜索

- **约束创造创造力**: "最终你应该添加尽可能合理的约束，以便工程师和设计师能够为任何你试图做的事情想出最有创意的解决方案"

## 4. 实战手册

**职业路径建议**:
1. **不要直接做PM** - 先从工程师、设计师或销售开始
2. **在这些角色中理解产品构建** - "他们是尽可能最小循环的最前线"
3. **找到你"可能世界级"的领域** - 不是"我永远成不了世界级工程师所以我就做PM"，而是"我在这些方面还可以，我可能在这个其他方面世界级"

**写作提升方法**:
1. **消费大量好的写作** - Anthony Bourdain："要懂如何做好食物，你得吃很多食物，并且愿意时不时吃顿坏饭"
2. **不要只读PM文档** - "你不会通过读PRD或支持文章成为更好的写作者。它需要是引人入胜的写作"
3. **关注节奏** - 打断单调模式，短-长、长-短句子交替
4. **找到引人入胜的写作** - Paul Graham essays、Twitter上的具体声音

**PM-工程-设计协作**:
1. **画边界（约束）**:
   - 客户细分："我们在服务坐下来餐厅还是taco truck？"
   - Jobs to be done：他们试图做什么？
   - 可用性：桌面web、移动web、原生移动
   - 原则：速度 vs 数据一致性

2. **关注最终交付**:
   - Kevin和设计师花整整一周微调菜单组弹出动画的毫秒数
   - 带入服务器和酒保测试原型，观察他们犹豫的地方
   - 推迟发布三次以确保质量

**决策日志实践**:
1. **每日日志** - 一天中发生的一切，用**hashtag decision**标记决策
2. **预测他人决策** - 看其他团队/公司，写下"如果我是负责人，我会做什么，为什么"
3. **设置提醒** - X周/月后检查结果
4. **从小开始** - 每周日早上花10分钟做一个预测

**反馈收集技巧**:
1. **Silent read（默读会议）** - 不说话，只读文档的20分钟专注时间
2. **实时评论和回应** - 减少延迟，加速周期
3. **调音叉方法** - 放置具体初稿让团队反应，而不是问抽象意见

**招聘创新**:
**Unsell Email** - 在offer阶段，发送邮件说所有可能强化他们恐惧的糟糕事情。"如果你能告诉他们 upfront，他们读完整个邮件仍然同样兴奋加入你，你就找到了A+招聘"

## 5. 反常识视角

- **PM不是必需的** - Stripe、Twilio、Figma等公司"从未有PM"是因为他们为自己构建产品 - "设计师为设计师，工程师为工程师"

- **移动慢以移动快** - 通过安排会议审阅文档来加速反馈，而不是异步"给我反馈"邮件

- **过度投资设计** - 大多数团队在设计上投资不足，"我更愿意雇佣增量设计师而不是PM"

- **决策日志不是构建的替代品** - 警告："如果你认为你可以只是坐在椅子上，展望市场，做一堆呼吁然后变得聪明，而不实际动手构建产品，你实际上不会变得更好"

- **销售误解** - "销售在科技中的声誉不好...人们只关心配额、数字等等。实际上最好的销售是最好的倾听者，最擅长理解客户的问题然后将其转化为你能为他们做的事情"

- **动画的重要性** - 花整整一周微调菜单组弹出动画的毫秒数，因为这影响了服务员的肌肉记忆和采用速度
