# Melissa Perri 播客摘要

## 嘉宾简介

Melissa Perri 是产品管理领域的传奇人物，《Escaping the Build Trap》和《Product Operations》两本 foundational 书籍的作者。她是 Produx Labs 的CEO，在哈佛商学院教授产品管理，通过 Product Institute 培训了将近**所有财富100强公司**的产品经理。她曾与4000多名产品经理合作，深度指导过30多家公司的产品转型，从高增长SaaS公司到银行、制药公司等传统企业。她的工作重点是帮助企业建立强大的产品管理功能，从战略制定到执行落地。

## 核心议题

本次对话的核心议题是 **产品管理在规模化过程中的常见挑战**，特别是当公司从找到产品市场契合点（PMF）转向规模化增长阶段时遇到的问题。Melissa 分享了她见过的最普遍的问题：**如何设定和部署战略**，以及为什么大多数组织的问题根源不在人员培训，而在目标设定和战略传达。她详细讨论了何时需要招聘首席产品官（CPO）、如何识别缺乏战略的信号、如何创建有效的产品愿景，以及为什么"产品运营"（Product Operations）角色正在成为产品管理领域最重要的变化。

## 核心洞察

### 规模化转型的关键节点
- 公司从单一产品转向多产品时，必须重新思考整个战略
- 从"找到PMF"到"规模化增长"的转变是最困难的阶段
- **关键问题**：如何优先化？如何制定综合战略？如何向数百上千名员工部署战略？
- 高增长公司通常在达到 **1000-3000万美元ARR**、**20-30名产品经理**时需要CPO

### 缺乏战略的明显信号
- 团队每周工作80小时，拼命工作但**指标没有移动**
- 产品对高管来说是"黑盒"——不知道团队在做什么，是否在实现目标
- **"缺失的中层"（missing middle）**：团队知道自己在做什么，高管知道公司愿景，但两者之间没有连接
- 战略存在于人们脑海中，但没有被写下来或清晰传达

### CPO与VP产品的根本区别
| VP产品 | CPO（首席产品官） |
|--------|------------------|
| 功能性领导，擅长实施流程 | 高管层成员，面向董事会 |
| 管理一到两个产品 | 管理复杂的多产品组合 |
| 执行导向 | 战略导向 |
| 可能不向设计汇报 | 通常向设计、产品运营、分析线汇报 |
| 专注于让产品工作顺利 | 深度理解财务，能从路线图预测收入 |
| 与CRO/CFO合作能力较弱 | 与CRO/CFO紧密合作，向董事会预测 |

### 战略部署的完整框架
1. **战略意图**（Strategic Intents）：2-3年的大方向（如：进军高端市场、地理扩张、创新进入新市场）
2. **产品举措**（Product Initiatives）：针对大问题的问题导向举措
3. **方案/选项**（Options）：具体要构建的解决方案
4. **团队工作**：实际执行的功能特性

**关键**：每一层都要有1-2页的文档说明"我们在做什么、为什么"，并且层层链接，形成战略树。

## 心法与原则

### 关于战略制定的核心原则

#### 1. 战略来自数据，而非猜测
"作为领导者，如果你从未制定过战略，你不知道从哪里开始。你需要从**所有地方的数据**开始。"
- 内部数据：财务系统、用户分析
- 客户研究：谁在与客户交谈？
- 公司目标：将背景信息整合

#### 2. 战略是关于选择，而非确定性
"好的领导者是做出当时尽可能最好的知情决策，然后如果发现错了，就愿意纠正自己。"
- 你不可能100%确定这会奏效
- 收集所有信息，做出最佳猜测，朝一个方向前进
- 不断重新评估，如果不对就转向

#### 3. 战略必须被清晰书写和传达
"我告诉人们，如果你们认为没有战略，去审问人们一会儿。问领导者：'如果我们达到这些数字是好事还是坏事？为什么？'"
- 战略不应该只存在于人们脑海中
- 每个团队都应该讲述相同的故事
- 从团队层面的战术工作一直回连到业务指标

#### 4. 执行是战略的关键
"99%的时间，我认为组织的问题实际上在于他们**设定目标和部署战略的方式**，而不是人员培训。"
- 一旦培训了人员，他们没有上下文知道朝什么方向努力
- 这是一个整体方法，转型时需要有人来执行

### 关于产品运营的洞察

"我意识到，当我们培训了所有团队，部署了战略，现在我们有这么多产品管理角色的人，然后你看某些事情，你意识到它**并没有规模化到其他团队**。"

产品运营解决三个问题：
1. **业务数据洞察**：让产品经理能够做出明智决策
2. **客户和市场洞察**：扩展用户研究，避免重复询问同一客户
3. **流程和实践**：标准化路线图格式、季度规划节奏、跨部门互动

## 实战手册

### 何时需要CPO？检查清单

**明确的信号**：
- [ ] 高管和董事会说："我不知道技术和产品部门在做什么"
- [ ] 产品经理很优秀但很沮丧，因为他们没有得到足够的方向
- [ ] 招聘产品经理困难，没有他们可以学习的人
- [ ] 执行团队之间内斗严重，目标不一致
- [ ] 产品经理不知道自己在构建什么以及为什么

**规模指标**：
- ARR达到1000-3000万美元
- 7-8名产品经理
- 多产品（2个以上）
- 地理扩张或进入新市场
- 重大转型或合并

### 如何创建有效的产品愿景

#### 第一步：愿景必须足够具体
"愿景应该足够具体，让人们能够在脑海中描绘出来。它不能是虚幻的，比如'成为医疗保健的骨干'。那是什么意思？我不理解。"

#### 第二步：回答关键问题
- 我们有何不同？
- 我们为谁服务？未来客户是谁？
- 我们提供的价值如何与今天不同？
- 我们提供价值的方式与竞争对手有何不同？为什么更好？

#### 第三步：说明我们不做什么
"我喜欢读到一个愿景说：'我们不会像那样。'这对我来说非常强大。"

#### 第四步：结合呈现和写作
"我喜欢结合呈现和写作。如果你能用设计师帮你，制作一些视觉效果，那就太棒了。"

### 如何制定战略：Melissa的方法

1. **收集所有数据**
   - 聘请数据分析师（前麦肯锡顾问很擅长）
   - 提出你想回答的问题，他们会获取数据、发现模式

2. **与理解市场、财务的人交谈**
   - 与销售交谈：人们为什么买竞争对手？为什么输？
   - 与其他部门交谈：他们有丰富的知识

3. **回答核心问题**
   - 我们如何获胜？
   - 我们如何进一步接近目标（愿景）？
   - 同时考虑我们现在的位置和我们现在能够执行什么

4. **做出选择并持续重新评估**
   - "我们采用所有信息，做出尽可能好的猜测朝一个方向前进，然后只是不断重新评估确保方向正确。如果不正确，我们就转向。"

### 如何部署战略：三层框架

**第一层：高管层（Executive Level）**
- 撰写两页文档：公司从哪里来？今天有什么不同？
- 外部威胁是什么？竞争对手是什么？
- 我们要关心什么？不关心什么？
- 我们要做什么？不做什么？
- **优先考虑战略意图**（未来2-3年的大业务推动力）

**第二层：产品领导层（Product Leadership）**
- 撰写产品举措文档
- 问题导向（围绕我们可以为客户解决的大问题）
- 通常由多个史诗（epics）组成

**第三层：产品团队层（Product Teams）**
- 定义解决方案/选项
- 回连到产品举措 → 战略意图

**关键**：每层1-2页，全部链接在一起，形成战略树。

### 产品经理如何提升战略能力

#### 如果还不是CPO：
1. **假装自己是CPO**
   - 你会做什么不同的事情？
   - 能深入挖掘数据吗？
   - 能问问题吗？

2. **与真正理解市场、财务的人交谈**
   - 你的CPO：你的流程是什么？如何设定？如何达到这三个优先级？
   - 销售部门：为什么人们买竞争对手？我们为什么输？

3. **分析其他公司如何制定战略**
   - Netflix如何制定战略？
   - 阅读关于公司如何从A到B的文章
   - 看到每个人做法不同，但都通过问这些问题达成

### 大型公司（银行、保险、制药）的产品转型

#### 常见问题：
- 采用SAFe（Scaled Agile Framework）但不理解何时应该偏离
- 产品所有者（Product Owner）被写成8,000个用户故事
- 没有时间与客户交谈（"我必须写用户故事"）
- 战略存在，但没有被清晰传达或部署

#### 转型方法：
1. **首先需要CPO**：在C-suite中代表产品
2. **混合人才**：保留一些人，也引入有经验的人
3. **培训所有人**（baseline），然后让他们练习
4. **让一些人选择退出**：意识到产品管理不是他们想要的
5. **构建职业路径**：产品经理应该能够晋升到管理多个团队、整个产品线

#### 职业路径建议：
- 产品经理 → 高级产品经理 → 总监 → VP → CPO
- **不要**将产品所有者和产品经理作为分开的职业路径
- 如果头衔是产品所有者，简历上要写产品经理/产品所有者
- 简历重点：**价值导向而非流程导向**
  - ❌ "我优先考虑积压工作，为开发者分解工作"
  - ✅ "我领导登录API工作，通过采访用户、与法律/合规团队合作，在满足监管要求的同时节省了数百万美元，为400万客户提供了无缝转换"

## 反常识视角

### 1. 大多数组织的问题不在培训，而在战略部署
"我见过很多组织认为他们的大多数问题在于人员培训。99%的时间，我看到这实际上在于他们设定目标和部署战略的方式。"
- 因为一旦你培训了这些人，他们没有上下文知道朝什么努力
- 这就是为什么当进行转型时，需要有人来做这件事

### 2. SAFe（规模化敏捷框架）的问题
"我不推荐使用SAFe。我与之交谈并喜欢SAFe的每一个人，他们发现成功后，最终都**把它撕碎**，改成其他东西。"
- 荷兰一家水务公司采用SAFe后破产，因为专注于流程而非为客户服务
- "如果流程不服务于公司、文化和客户的最佳利益，就修复它、改变它"
- SAFe的创始人Dean Leffingwell从Rational统一过程演变而来
- **问题**：过度关注流程，忽略战略、产品管理和价值交付

### 3. 敏捷宣言的签署者中没有产品经理
"没有一个人签署敏捷宣言的是产品经理。我花了很多时间与这些人交谈。"
- 敏捷是软件开发者发起的运动
- 产品所有者角色从Scrum产生，而不是从产品管理
- Scrum指南2013版本删除了"产品所有者可能是产品经理"的说明

### 4. 大型技术公司（Google、Amazon、Netflix）没有产品所有者
"我基本上不知道任何大型技术公司有产品所有者。他们全是产品经理。"
- SAFe主要在银行、电信、制药等非软件原生公司中采用
- 这些公司需要"手册"来指导如何做软件
- **问题**：将产品管理的责任视为"团队角色"而非"业务角色"

### 5. 产品运营不是外包决策权
"产品运营不会剥夺产品经理的决策权。它是为了告知他们。"
- 产品运营是"产品经理的产品经理"
- 如果产品运营说"你应该构建这个"，那不是他们的工作
- 产品经理仍然负责：战略、愿景、优先级、资源分配、权衡决策、艰难的利益相关者对话

### 6. 产品运营团队应该很小
"如果你是一对一的比例，你就做错了。绝对错了。"
- 不要雇佣10,000名产品运营人员
- 应该是一个精干、高效的团队
- 利用工具和软件来自动化流程
- 产品运营的人数取决于公司数据工具化程度

- 目标：在三个月内雇佣全职CPO，保持足够的路线图让团队继续交付

### 对认证的看法
"我会认证产品经理吗？不。我认为你不能从短班中让人准备好并能够做他们的工作。"
- 敏捷认证通常只是2天的课程
- 敏捷行业复杂体通过认证和咨询赚钱
- 她的课程提供"完成证书"，而不是"能力证书"

### 常见的战略设定错误
一家医疗保健公司的愿景："成为医疗保健的骨干"
- Melissa问："那是什么意思？"
- 没有人能够详细说明
- **"那是个标语，不是愿景。"**
- 没有人说我们在实现什么？我们不做什么？我们长大后是谁？

### 首席产品官的关键问题
当咨询CPO时，Melissa会：
- 去所有团队问："你在做什么？你能做的最重要的事情是什么，为什么？"
- 尝试自己将其连接成战略，看是否相关
- 如果不相关 → 没有人制定和部署战略
- 问："这里有CPO吗？如果有，这个人可能不适合这个工作。如果没有，我们需要有人来做这件事。"

### 产品管理职业化的重要性
"我们把这个称为业务角色是有原因的。它一直贯穿到帮助你进一步发展你的业务。"
- 必须确保团队上的人可以被晋升到运行多个团队、运行整个产品线
- 在硅谷原生软件公司中这很简单，但在其他组织中 unheard of

### 与敏捷教练的区别
"我们看到这个趋势，人们把敏捷教练扔到这个。"
- 敏捷教练从未在运行良好的公司担任过产品经理
- 他们会恢复到敏捷流程，优化Scrum之类的东西
- **不是**帮助制定产品管理流程

### 对产品经理的建议
"作为一名产品经理，即使你已经在这个角色上工作了一段时间，最重要的是确保你总是在学习。但你会学到最多的通常是**通过执行**。"
1. 首先专注于做你的工作
2. 分析你的工作：什么有效？什么无效？
3. 选择你想改进的部分，深入研究
4. 创造你自己的工具箱并继续前进

### 关于敏捷流程的最终建议
"如果它不为你服务，就继续前进。改变它。一切都是要迭代的。一切都是要适应的。如果不为你工作，你不必继续这样做。"
- Scrum或任何流程都是如此
- 坐下来，与自己做回顾，说："这让我成为更好的产品经理吗？"
- 如果不是，改变它。改变你的方法。做点不同的事情。
