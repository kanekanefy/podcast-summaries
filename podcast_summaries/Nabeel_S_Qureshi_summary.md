# Nabeel S. Qureshi - Lenny's Podcast 摘要

## 嘉宾简介

Nabeel S. Qureshi 是创始人、作家、研究员和工程师，曾在Palantir工作近8年，担任Forward Deployed Engineer，负责与美国联邦机构的公共卫生项目合作，包括COVID-19应对和药物发现中的AI应用。他也是GoCardless（欧洲最大的金融科技独角兽之一）的创始成员和商务开发副总裁。

## 核心议题

1. **Palantir的独特文化** - 如何通过极端的价值观筛选创造创始人孵化器
2. **Forward Deployed Engineer模式** - 深度客户现场工作的革命性产品开发方法
3. **从服务到产品的转型** - 如何将定制化解决方案转化为可扩展平台
4. **数据平台的秘密** - 数据整合、清理和分析的隐藏价值

## 核心洞察

### 1. Palantir的创始人工厂现象
- **30%的离职PM创建公司** - 远超其他公司（第二名Intercom仅18%）
- **YC中Palantir校友创始人超过Google** - 尽管Google规模大50倍
- **成功的关键筛选机制**:
  - 独立思考者（敢于挑战权威）
  - 广泛的知识兴趣（引用Habermas等哲学家）
  - 极度竞争心态（"不惜一切代价获胜"）

### 2. Forward Deployed Engineer (FDE) 模式
**核心特征**:
- 每周周一到周四在客户现场办公
- 在客户公司有专属办公桌
- 深度融入客户日常工作流程

**为什么有效**:
```
传统模式: 偶尔访谈 → Zoom会议 → 分享Mock
FDE模式:   现场办公 → 建立信任 → 深度理解 → 快速迭代
```

**反馈循环速度**:
- 周一: 会议发现问题
- 周一晚: 构建解决方案
- 周二: 展示并获取反馈
- 周二晚: 迭代改进
- 每周4-5个循环 vs 传统模式每月1个循环

### 3. "Save the Shire" - 独特的人才吸引策略
**时代的对比** (2010年代):
```
热门公司: Facebook, Pinterest - "社交、移动、本地"
Palantir:  "保护西方" - 军事、国防、情报
```

**刻意设计的"坏信号"**:
- 不吸引所有人才能吸引最合适的人
- 清晰定义"这不适合谁"比"这适合所有人"更重要
- 类似今天的OpenAI/Anthropic对AI救世主式使命的强调

### 4. 从Consulting到Product的转型之路
**关键发现**:
- 数据整合是大型组织中最痛苦的问题
- 95%的工作是数据访问、清理、整合 - 分析只是冰山一角
- 内部工具产品化 → Foundry诞生

**Airbus案例 - "Asana for making planes"**:
- 问题: A350产量需要4倍增长
- 核心困难: SAP数据难以理解，工作流程不透明
- 解决方案: 将数据库表映射为人类可理解的概念
- 成果: Ontology成为Foundry核心组件，4倍提升生产效率
- 通用化: 特定解决方案 → 平台级产品

### 5. 无头衔文化的哲学
**Thiel的洞察** (来自《Zero to One》):
- 头衔成为竞争对象 → Goodhart's law生效
- 人们开始优化指标而非创造价值
- Palantir方案: 所有人都是"Forward Deployed Engineer"

**效果**:
- 角色基于能力而非头衔
- 始终需要证明自己配得上当前工作
- 避免"为了晋升而启动新项目"的Google式问题

## 心法与原则

### 对产品管理的非传统观点
**Palantir PM的特质**:
1. **必须先成为FDE** - 不能直接从外部雇佣PM
2. **从实战中选拔** - 在现场证明过客户同理心和执行力
3. **工程师思维** - 要么是工程师出身，要么与工程师关系极好

**避免"Google Docs综合症"**:
- 拒绝PRWD驱动的理性管理
- 拒绝脱离现实的文档工程
- PM与工程师必须建立深度信任

### 创始人训练场
**为什么FDE培养创始人**:
1. **销售体验**: 见识大额交易（百万美元级别）
2. **问题发现**: 深度理解客户真实痛点
3. **产品构建**: 从零构建解决方案
4. **迭代速度**: 每周多次反馈循环
5. **责任担当**: 对结果负责，不是"只是建议"

### 道德框架 - 如何看待有争议的工作
**核心观点**: "脱离接触很少是正确答案"

**Google Maven项目的反例**:
- 员工抗议国防项目 → Google退出
- 这偏离了美国主流观点
- 结果: 错失影响力机会

**在房间里改进的论点**:
```
2010年方式 vs 现代Palantir方式
更精确 → 更少附带伤害 → 更少错误
在房间里改进 vs 在外面批评
```

**技术的政治性**:
- 2025年: 科技已深度介入政治
- Mark Zuckerberg在国会听证会的画面
- 工程师必须思考: 优化短视频对儿童大脑的影响

## 实战手册

### Forward Deployed Engineer模式实施指南
**何时适用**:
- 客户年收入必须达到数十亿美元级别
- 交易规模足够大（数百万美元）以支持现场投入
- 问题复杂且需要深度理解

**调整版本** (针对较小公司):
- 1个FDE服务5个客户（而非全职1对1）
- 减少现场时间但保持面对面接触
- 仍需技术能力和深度业务理解

**成功要素**:
1. **真正的工程师** - 能自己构建产品，不只是解决方案架构师
2. **现场工作** - 面对面建立信任
3. **深度关系** - 与客户成为朋友，理解业务动力
4. **业务知识** - 如医院如何运营（周转病床如餐厅周转桌子）

### 从客户项目到产品的转化
**错误模式**: 过于固守产品愿景
```
❌ "我们是分析软件，客户不关心内部分析"
✅ "客户有这个巨大问题，我们应该转向解决它"
```

**Retool案例**:
- 最初定位: "超级Excel" - 无吸引力
- 改变定位: "轻松构建内部工具" - CTO们纷纷回应
- 同一产品，不同框架，完全不同结果

**判断标准**:
```
何时坚持产品愿景:
- 找到更匹配的客户
- 问题确实存在但需要更好解释

何时转向客户问题:
- 问题足够大且紧急
- 有足够多其他客户有类似问题
- 团队有独特优势解决它
```

### Palantir式招聘
**寻找"多出20%"的人**:
- 不是完成JD要求的人
- 是会主动推动结果达成的人
- 大公司 vs 创业公司的思维差异

**筛选问题**:
- "你为达成某事最努力的一次是什么？为什么？"
- 寻找:
  - 使命契合（个人经历驱动）
  - 过往的极致努力经历
  - 非技能相关的信号

**"坏信号"的艺术**:
- Max Levchin的篮球故事
- 如果喜欢打篮球 → 立即拒绝
- 信号: 不是超级Linux极客 → 不适合PayPal早期文化
- 目的: 明确谁不适合，从而吸引最合适的人

### 快速迭代原则
**核心公式**:
```
成功 = 下注数量 × 单次成功率
```

**实施建议**:
1. **早期验证**: 尽快向客户展示并要求付费
2. **快速失败**: 不要等3周，3天足够判断
3. **大量下注**: 提高尝试频率

**Nabeel的原则**:
- "EOP成功与下的赌注数量成正比"
- 测试非常早，客户愿意付大钱就继续，否则找新问题

## 反常识视角

### 1. 大学教育的价值
**科技界的反共识**:
```
主流观点: 除非特殊情况，否则退学创业
Nabeel:   大学是宝贵的，不要轻易退学
```

**大学的价值**:
- 唯一可以深度交友的时期
- 专注思考和阅读的时间
- 尝试不同事物，发现自己
- 21岁后很难再有这种时间和环境

### 2. 竞争 vs 协作文化
**无头衔的真实效果**:
- 表面: 避免头衔竞争
- 实际: 竞争转向获取特定人物的青睐
- 但仍有价值: 始终需要赢得工作权

### 3. 数据平台的核心竞争力
**表面**: 分析工具
**实际**: 数据整合和清理
**冰山理论**:
```
可见部分 (5%): 分析、仪表盘、可视化
隐藏部分 (95%): 访问、清理、整合、规范化
```

### 4. 最大的竞争对手
**不是**: 其他数据平台公司
**而是**: 客户自己构建解决方案
**CIO的思维**: "我要自己拥有和控制数据基础设施"
