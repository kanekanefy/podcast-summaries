# Nan Yu (Linear Head of Product) - Lenny's Podcast 摘要

## 嘉宾简介

Nan Yu是Linear的产品负责人，Linear是目前最受欢迎、设计最美观、增长最快的B2B SaaS产品之一。他曾在Everlane（直接面向消费者的服装品牌）和Mode（商业智能工具）工作，在竞争激烈的行业中积累了丰富的产品经验。

## 核心议题

1. **速度与质量的统一** - 打破"速度vs质量"的权衡迷思
2. **避免产品臃肿** - 如何拒绝会导致产品恶化的功能请求
3. **情感驱动的产品设计** - 寻找用户"感觉糟糕"的时刻并解决它
4. **系统化的创造力** - 通过测试极端版本来找到最佳解决方案
5. **B2B软件的教育功能** - 软件不仅是工具，更是工作方式的教学

## 核心洞察

### 1. 速度与质量并非对立面
**主流错误认知**:
```
人们认为：速度 = 匆忙/草率
真相：速度 = 能力/专业
```

**Patrick Collison的推文**:
> "我越来越相信'好、便宜、快——选两个'这个格言是慢的人散布的恶意误导。在我的经验中，慢和昂贵通常相伴而生。"

**Magnus Carlsen的例子**:
- 他是国际象棋世界冠军
- 也是快棋世界冠军
- 顶级选手在极短时间内仍能做出高质量决策
- 速度本身是能力的体现，不是草率的证明

### 2. Linear的速度原则
**10%法则**:
```
项目时间预算: 1个月
→ 第1周（10%时间）: 必须有可工作的解决方案
→ 目标: 验证关键假设，而非达到完美
```

**为什么大多数团队做不到**:
- 过度追求第一版的完美
- 需要像素级完美设计
- 需要修复所有UI小bug
- Linear的态度: 第一版只是最佳猜测方向

**渐进式用户测试**:
```
内部团队 → Beta客户群 → 扩大Beta用户 → GA全面发布
```

### 3. 防止产品臃肿的核心原则
**Linear的绝对拒绝**:
> "我们绝对不会做的事情是：中层经理为了更容易汇报而要求的定制化功能，这些功能会让IC（个人贡献者）的工作流程变得更糟。"

**两类用户的激励冲突**:
```
中层经理需求:
- 想要详细报告
- 想要追踪一切
- 愿意为这些功能买单

IC (工程师/设计师) 需求:
- 想要快速完成工作
- 不想填写额外字段
- 绩效评估基于代码贡献，非填写工单

IC的行为:
→ 随机选择下拉选项
→ 祈祷老板不要用这些数据做决策
→ 结果: 所有报告数据都是错误的
```

**Linear的承诺**:
- 不做这个权衡（trade-off）
- 一旦走上这条路，IC用户体验会恶化
- IC会脱离使用，数据质量下降
- 最终让所有人都不满意

**如何应对销售压力**:
- 买家（中层经理）会有功能清单
- 前3个需求通常是最重要的
- 如果能解决前3个需求且质量远超竞品
- 其他7个需求可以协商
- 证明这些功能反而会导致当前混乱局面

### 4. 深度挖掘用户需求的技巧
**Nan的目标**:
> "我的目标是感受与客户相同的不愉快感。"

**超越传统的"5 Whys"**:
```
传统方法:
→ 客户想要X
→ 为什么？
→ 为了达成目标Y
→ 作为角色Z...
→ (停留在功能层面)

Nan的方法:
→ 客户想要X
→ 如果没有X会怎样？
→ 我升职无望
→ 现在我理解了问题的严重性
→ (到达情感层面)
```

**真实案例 - 日期问题**:
```
客户表面需求:
→ "营销和我永远无法对齐"

深层挖掘:
→ 客户分享了具体故事
→ "我把项目日期定为12月30日（Q4项目）
   想放在最后
→ 营销团队疯了
→ '12月30日大家都在休假，怎么发布？'"

情感真相:
→ 这种糟糕的沟通不断重复
→ 再也不想设定具体日期了

Linear的解决方案:
→ 允许灵活的目标日期（12月、Q4、2024下半年）
→ 不必给错误的精确度
→ 避免误沟通
```

### 5. 系统化的创造力方法
**核心问题**:
```
大多数人的困境:
→ 能看到眼前的事物
→ 但无法推演2-3步后的可能性
→ 选项太多，不知往哪个方向走
→ 最终在平庸的选项中选择
```

**Linear的解决方案 - 测试极端版本**:
```
1. 选择一个属性（如速度）
2. 问：这个属性最极端的版本是什么？
3. 构建最极端版本
4. 测试它
5. 学习并找到平衡点
```

**草稿保存功能案例**:
```
版本1: 最快体验
→ 关闭时不询问
→ 不保存任何确认
→ 结果: 感觉极其不安全
→ (只内部测试)

版本2: 最安全体验
→ 输入一个字符就自动保存
→ 结果: 留下一堆"无标题草稿"
→ 造成了混乱的痕迹
→ (推向用户测试)

最终解决方案:
→ 新issue创建时关闭 → 询问是否保存
→ 已存在的草稿编辑 → 自动保存
→ 非常具体的平衡
```

**为什么这有效**:
- 让你看到默认约束之外的可能性
- 避免在错误选项中选择
- 找到第四个隐藏在角落的正确选项
- 最佳解决方案总是事后看来显而易见

### 6. B2B软件的隐藏功能
**洞察**:
> "B2B软件不只是解决人们的问题，它还在教人们如何工作。"

**软件的起源**:
```
某公司内部人士
→ 实现了某个流程
→ "这流程对我们很有效"
→ 构建内部工具让同事也能用
→ 同事按下按钮，好事发生
→ 将流程和工具转化为创业公司
```

**购买软件 = 采用工作方式**:
```
营销软件例子:
不仅获得: 发邮件的能力
同时获得:
→ 组织活动（Campaigns）
→ 测量点击率
→ 计算获客成本
→ 这些都是"正确做法"
```

**ERP的极端例子**:
- 最痛苦的软件采用过程
- 需要重新设计所有内部流程
- 库存管理方式都要改变
- 公司愿意这样做因为知道这是"经过验证的最佳实践"

**Linear的观点**:
- 不是"武断的意见"
- 而是找到"很多高效团队的实际共识"
- 给不会这些实践的团队一个按钮
- 让他们也能开始这样做

## 心法与原则

### 寻找"情感钩子"
**Paul Graham的"Schlep Blindness" (盲目劳累)**:
```
你的日常:
→ 讨厌周一
→ 为什么？
→ 周一要花时间收集信息写报告
→ 如果有按钮能自动生成报告
→ 可能就不那么讨厌周一了

你需要外部观察者:
→ 注意你情感变化的节奏
→ 找到可以改进的点
→ 这些是别人没注意到的机会
```

**Triage Management功能的诞生**:
```
两类人的痛苦:
类型A (手动管理):
→ "我在水下快窒息了"
→ "要处理这么多工单，正确路由"
→ 感觉很糟糕

类型B (不管理):
→ "完全失控"
→ "人们把工单扔过墙"
→ "不知道它们在哪里"
→ "没有期待会得到回应"

共同根源:
→ 没有自动化、有组织的队列管理方式
```

### 现实 vs 优雅
**PM的陷阱**:
```
错误做法:
→ 构建美丽优雅的解决方案
→ 不问"现实也是美丽优雅的吗？"

真相:
→ 现实有时很丑陋
→ 美丽但不匹配现实的方案
→ 没有长期生命力
```

**Linear的做法**:
- 将功能需求回溯到具体的人
- 不是"Alice"或"Bob"这样的假设人物
- 而是"FirstName LastName, Email可查"
- 可以实际联系验证的人

### 迭代过程 - "Kneeling"
**Nan的术语**:
```
你有东西
→ 形状不完全正确
→ 投放到野外
→ 收到反馈"不太匹配现实"
→ 问: "我们测试过这个方面吗？"
→ "我们是否正确匹配了现实？"
```

**两类反馈策略**:
```
类型1: 我们想错了
→ 不需要大量反馈
→ 不是数量问题
→ 而是"我们想对了吗？"

类型2: 大功能请求
→ 很多人要
→ 100种不同用例
→ 选择: 构建覆盖所有用例的大功能？
→ 还是找到集中的用例池，做得特别好？
```

## 实战手册

### Double Triangle组织模式
**传统三角** (常见说法):
```
PM + 工程师 + 设计师协作
```

**Linear的Double Triangle**:
```
构建侧:
→ PM
→ 工程师
→ 产品设计师

销售侧:
→ PM (在中间)
→ 销售
→ 产品营销

PM的角色:
→ 连接构建侧和销售侧
→ 将商业目标转化为产品
→ 用工程可能性来调节
```

**为什么这重要**:
- PM是"面向市场的产品管理学科"
- 很多组织产品与销售/营销有对立
- Linear紧密合作：
  - PM帮助营销团队选对措辞
  - 让他们听起来像本地人
  - 工程师和PM有敏感的BS探测器
- PM团队有全职产品营销人员
- PM团队产出来源：
  - 所有变更日志
  - 发布说明
  - 即将发布的功能语言

### 如何找到工作 - PM求职策略
**Nan的独特方法**:
```
传统PM面试:
→ 展示自己是全能的
→ 缺点很少
→ 与其他候选人无差异

Nan的方法:
→ 发现招聘经理的"待解决问题"
→ 让自己成为"解决方案"
→ 招聘变成二选一:
  - 雇佣问题的解决方案？
  - 还是雇佣其他人？
```

**"假装你已经在那儿工作"**:
```
在面试中:
→ 面试官: "你有什么问题吗？"
→ 你: "你们本季度的OKR是什么？
       我如何帮助达成这些？"
→ 结果: 获得具体的智能信息
→ 而非一般性的高层面问题
```

**深入挖掘**:
```
1. 从面试流程中的人开始
2. 做发现性问题
3. 如果想深入某个领域
4. 询问: "能让我和解决同一问题的工程经理聊聊吗？"
5. 如果没别人问这个
→ 额外的反馈点: "这个人问的问题真好"
→ 在debrief过程中脱颖而出
```

### 截止日期的管理
**Nan的观点**:
> "不要有太多截止日期，但当你有截止日期时，它是P0级别的。"

**工程师vs截止日期的常见认知**:
```
工程师观点:
→ "截止日期完全是虚构的"

让截止日期成真的唯一方法:
→ 认真对待它到P0级别
→ 其他所有事都不重要
→ 向团队和利益相关者发出真实信号
```

**PM在截止日期的角色**:
```
1. 尽可能削减范围
2. 找出真正阻碍的东西
3. 目标: 在go/no-go决策时
   能对产品说"是"或"否"
   (而非产品还差太远，无法决定)

4. 产品可能没有你想要的所有功能
   但它是可以实际发布的真实产品
```

**何时值得有截止日期**:
- 通常与外部营销活动相关
- 涉及某种外部营销类型练习

**为什么错过发布机会是真实的损失**:
```
与客户沟通的机会是有限的:
→ 一年365天
→ 12个月
→ 每月约4周
→ 每周有机会向受众说一次话
→ 或12个月说一次大事
→ 或4个季度说一次大事

错过一个机会:
→ 无法时间旅行回去重做
→ 无法回来说"实际上，让我们重做Q1"
```

**Linear如何应对截止日期**:
- 几乎不做估算来达到截止日期
- 方法：尽早发布
- 10%时间有可工作版本
- 剩余时间决定：再次迭代？还是打磨到可发布状态？
- 为未来的自己创造可以决策的位置

## 反常识视角

### 1. "好、便宜、快——选两个"是谎言
**真相**:
- 慢和昂贵通常一起出现
- 能力强的人既快又好
- 装修房子的例子：工期长 = 住酒店 + 账单累积

### 2. 大学教育的价值
**虽然Nan没有明确反对退学**，但整个对话暗示了深度学习和经验积累的价值。他在竞争激烈的行业（时尚、BI、项目管理）都取得了成功。

### 3. 定制化请求的陷阱
**客户说**:
- "我需要这10个功能"

**真相**:
- 前3个最重要
- 如果能通过原生功能解决前3个且质量远超
- 其他7个可以协商
- 通过"可视化编程"定制永远达不到Native功能的质量和深度

### 4. 工程师文化的重要性
**B2B软件的目标客户**:
- 工程师有敏感的BS探测器
- 知道什么是好的工作
- 如果PM说错话，会给臭脸
- 例如：不要称他们为"Project Managers"

### 5. 产品营销是PM的职责
**Airbnb的做法**:
- 每个PM也是PMM
- 不再有单独的产品营销经理头衔

**Linear的做法**:
- PM团队有全职PMM
- 但PM深度参与内容创作
- PM比任何其他团队更了解客户
- 应该帮助营销团队创造措辞

### 6. 正确数量 = 过多减一
**Nan的生活格言**:
```
例子: 吃多少披萨？
→ 5片太多，感觉不舒服
→ 4片正好
→ 要找到正确数量
→ 有时需要冲到边缘
→ 找出什么是"太多"
→ 然后就知道正确数量是什么
```

**与Elon Musk削减原则的联系**:
- 如果不恢复10%削减的东西
- 说明你砍得不够

**起源**:
- 一批有缺陷的男士T恤
- 都短了1.5英寸
- 不能销售（肚脐会露出来）

**解决方案**:

- 再剪掉2英寸，做成超短款

**结果**:
- 一周内售罄
- 现在有20种颜色
- 仍在销售
- 很难说是营销胜利还是设计胜利
- 以最奇怪的方式找到了产品市场契合
